# Alliance Docs 自研系统设计（弃用 XWiki 版）

## 1. 主公新决策（先说结论）

- 彻底放弃 XWiki 路线，不再依赖外部大系统；
- 我们只保留自研的 `alliance-docs`，做到可控、可改、可演进；
- 目标是做一套“团队文档协作工具”，必须支持：
  - 本地文件（导入/导出/同步）
  - Markdown 文档（编辑、预览、版本）

## 2. 两轮思考（大白话）

### 第一轮：为什么要弃用 XWiki

- XWiki 功能很全，但系统太大，很多能力不是我们当前阶段刚需；
- 配置、升级、插件链路都偏重，团队会被平台节奏绑住；
- 我们要的是“能快速按业务改”，不是“先学会一套庞大平台”。

### 第二轮：自研怎么避免翻车

- 不一次做全家桶，先做可用 MVP：文档+Markdown+本地文件；
- 采用分层架构：前端编辑层 / 本地同步层 / 协同服务层；
- 所有复杂能力按开关渐进上线（实时协同、权限、审计、AI）。

## 3. 已执行变更（今天）

- 已移除 XWiki 相关目录与配置（源码、部署、环境文件、旧文档）；
- 当前项目核心目录收敛为：
  - `/Users/project/notes/alliance-docs`（前端应用）
  - `/Users/project/notes/docs/alliance-docs`（系统文档）

## 4. 目标系统蓝图（0-1 到 1-10）

### 4.1 分层结构

1. 客户端（已存在）
   - React + TypeScript + Tiptap
   - 负责编辑、预览、搜索、状态看板
2. 本地文件桥接层（下一步）
   - 通过桌面壳（Tauri/Electron）或本地代理服务
   - 负责读写本地 Markdown 文件、监听文件变更
3. 协同服务层（第二阶段）
   - 提供账号、权限、版本、评论、多人协同
   - 存储可落 PostgreSQL + 对象存储

### 4.2 为什么这样拆

- 前端先保证“单人好用”；
- 文件桥接层保证“本地文件真可控”；
- 协同服务层保证“团队协作可扩展”。

## 5. 核心功能设计（按优先级）

### P0（必须）

- Markdown 文档编辑/预览双模式；
- 本地目录绑定（例如绑定 `~/team-docs`）；
- 自动保存到本地 `.md`；
- 文档列表、标签、全文检索；
- 基础历史版本（本地快照）。

### P1（应有）

- 文档冲突检测（同一文件被外部改动时提示并合并）；
- 协同评论与评审状态流；
- 文档权限模板（读/评审/编辑）。

### P2（增强）

- 实时多人协同（CRDT）；
- 企业登录（SSO）与审计日志；
- AI 摘要、变更说明自动生成。

## 6. 实现细节（重点）

### 6.1 Markdown 结构

- 存储规范：每篇文档一份 `.md` 正文 + 可选元数据头（frontmatter）；
- 页面展示时解析为富文本；编辑完成再回写 Markdown；
- 统一规则：标题层级、代码块、表格、引用、任务清单都要可逆转换。

### 6.2 本地文件能力

- 目录扫描：启动时递归读取绑定目录；
- 文件监听：检测新增/修改/删除并实时刷新侧边栏；
- 安全限制：只允许访问用户显式授权目录；
- 失败兜底：写入失败保留内存草稿并提示手动导出。

### 6.3 版本与冲突

- 每次保存生成本地快照（时间戳 + 文档 hash）；
- 如果文件在外部被改，进入“冲突态”：
  - 左侧：磁盘版本
  - 右侧：当前编辑版本
  - 提供按段合并并重新保存。

### 6.4 团队协作接口（预留）

- 文档实体统一 ID，不依赖文件名；
- 本地文件路径只作为映射字段，后续可迁移到远端仓库；
- 预留事件总线：`doc.created`、`doc.updated`、`doc.reviewed`。

## 7. 数据模型（大白话）

- `DocumentMeta`：文档身份证（id、title、owner、status、tags、更新时间）
- `DocumentBody`：正文（markdown、html-cache、word-count）
- `FileBinding`：本地映射（docId、absolutePath、mtime、hash）
- `VersionSnapshot`：版本快照（docId、version、source、createdAt）

## 8. 4 周落地节奏

- 第 1 周：Markdown 双向转换 + 文档规范
- 第 2 周：本地目录绑定 + 文件监听 + 自动保存
- 第 3 周：版本快照 + 冲突合并页
- 第 4 周：评审流 + 稳定性测试 + 使用手册

## 11. Go 后端与文件上传 MVP（2026-02-28）

### 11.1 主公本次指令

- 后端语言定为 Go；
- 直接开始实现上传链路，不再空谈。

### 11.2 小赵两轮思路

- 第一轮：先做“能跑通附件上传”的最小闭环，避免一上来就陷入账号/权限全家桶；
- 第二轮：采用“前端拿签名直传 MinIO，后端只做签名与元数据”的结构，上传效率高、后端压力小。

### 11.3 本次落地结构

新增目录：`/Users/project/notes/alliance-docs/backend`

关键文件：

- `cmd/server/main.go`：后端启动入口与优雅停机；
- `internal/config/config.go`：环境变量配置读取；
- `internal/httpapi/server.go`：路由、CORS、中英文错误响应、上传相关接口；
- `internal/storage/minio.go`：MinIO 客户端、桶初始化、上传/下载签名 URL；
- `internal/store/postgres.go`：PostgreSQL 连接与表初始化；
- `internal/store/attachment_repo.go`：附件元数据 CRUD；
- `docker-compose.dev.yml`：本地依赖服务（PostgreSQL + MinIO）。

### 11.4 已完成接口

1. `GET /api/v1/health`
   - 健康检查。
2. `POST /api/v1/uploads/presign`
   - 生成 MinIO 预签名上传 URL。
3. `POST /api/v1/uploads/complete`
   - 校验对象已上传，再把附件元数据写入 PostgreSQL。
4. `GET /api/v1/attachments/:id`
   - 查询附件元信息。
5. `GET /api/v1/attachments/:id/download-url`
   - 生成下载预签名 URL。
6. `GET /api/v1/documents/:documentId/attachments`
   - 查询某文档下的附件列表。

### 11.5 实现细节（大白话）

- 上传不是“把文件先传给 Go 再中转”，而是前端直接传 MinIO；
- Go 后端负责两件事：
  - 发签名（谁能传、传哪、多久有效）；
  - 记账（附件属于哪个文档、大小、类型、负责人）；
- 数据库存一张 `attachments` 表，字段覆盖文档 ID、对象键、文件名、大小、类型、创建时间；
- 服务启动时会自动检查并创建 MinIO bucket，不用手工点控制台。

## 12. 下一阶段作战规划（主公拍板版）

### 12.1 两轮推演结论

- 第一轮（先活下来）：先把“上传 + 文档持久化 + 基础权限”打透，确保团队能稳定使用；
- 第二轮（再做强）：再上实时协同、评论流、版本回滚，避免一次性做太重导致延期。

### 12.2 四阶段排期（建议 4 周）

1. 第 1 周：上传链路前后端打通（P0）
   - 前端接入 `presign -> PUT -> complete`；
   - 文档页可展示附件列表与下载；
   - 验收：可上传、可下载、可追溯到文档。
2. 第 2 周：文档后端化与登录（P0）
   - 文档元数据从 localStorage 迁到 PostgreSQL；
   - JWT 登录、刷新令牌、基础用户表；
   - 验收：换浏览器仍能看到文档与附件。
3. 第 3 周：权限与评审流（P1）
   - 文档角色（读/评审/编辑）
   - 状态流（草稿/评审中/已发布）服务端校验；
   - 验收：不同角色操作受控。
4. 第 4 周：协同增强（P1）
   - 评论与提及通知；
   - 历史版本快照与回滚；
   - 验收：可按版本回退，评审闭环可追踪。

### 12.3 风险与兜底

- 上传失败：前端保留待重试队列；
- 数据冲突：以服务端版本号做乐观锁；
- 迭代超时：P2（实时协同）一律后置，不拖 P0/P1。

### 12.4 里程碑验收标准

- M1：附件上传下载稳定（成功率 > 99%）；
- M2：文档与附件跨设备一致；
- M3：权限与评审流程可用；
- M4：版本回滚和评论流上线。
